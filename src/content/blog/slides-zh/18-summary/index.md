# 快速和慢速系统调用
* 在第一周的课程中，我们学习了称为**系统调用**的内核驻留函数。
    * 快速系统调用是那些立即返回的调用，"立即"意味着
      它们只需要处理器和其他本地资源来完成工作。
        * 根据我的定义，对它们允许花费的时间没有硬性限制。
        * 即使一个系统调用需要 60 秒才能完成，如果所有 60 秒都花在执行代码上（即没有
          因等待外部资源而空闲等待），它也被认为是快速的。
    * 慢速系统调用是那些无限期等待某事完成（例如 `waitpid`）、
      等待某事变得可用（例如从最近没有看到任何数据的客户端 socket 进行 `read`），
      或等待某个外部事件（例如通过 `accept` 接收来自客户端的网络连接请求）的调用。
        * 如果从本地文件读取，对 `read` 的调用被认为是快速的，因为没有真正的外部资源
          阻止 read 完成其工作。虽然确实需要访问一些硬件，但因为该硬件是嵌入在机器中的，
          我们可以相当有信心地说，从本地文件读取的数据将在一定时间内可用。
        * 如果数据被发布到 socket 并且之前发布的数据已经拥塞了内部缓冲区
          并且尚未被推送到机器之外，则对 `write` 的调用被认为是慢速的。
    * 慢速系统调用是那些能够无限期阻塞执行线程的调用，
      使线程在系统调用能够返回之前处于非活跃状态。
        * 我们依靠信号和信号处理程序将对 `waitpid` 的调用从正常执行流程中移出，
          我们还依靠 `WNOHANG` 标志来确保 `waitpid`
          永远不会真正阻塞。
            * 这就是非阻塞的一个例子。
            * 只是我们当时没有这样称呼它。
        * 我们依靠多线程将对 `read` 和 `write` 的调用
          从主线程中移出。（注意：请记住 `iosockstream` 类实现在
          对 `read` 和 `write` 的调用之上分层。）
            * 线程化并不能使对 `read` 和 `write` 的调用更快，
              但它可以并行化停顿时间，并且释放主线程，使其可以（如果选择的话）专注于其他计算。
            * 基于你在 `news-aggregator` 和 `proxy`
              上的工作，你应该对这些思想非常熟悉。
    * `accept` 和 `read` 是大家通常
      标识为慢速的系统调用。

::: tip 重难点解析
**快慢的判定标准 —— CPU 时间 vs 墙钟时间**：区分"快速"和"慢速"系统调用的关键不在于它们总共耗时多久，而在于线程是否因为等待外部事件（网络数据到达、子进程结束、磁盘寻道等）而被挂起。一个计算大量数据的系统调用即使花了 10 秒，期间 CPU 一直在全速运转，它也是"快速"的。`read` 一个 socket 很可能只花几毫秒就返回了数据，但如果数据还没到达，线程就会无限期阻塞，所以它是"慢速"的。这个区分的实际意义在于：我们需要用不同的策略来处理这两类调用——对于慢速调用，需要信号、多线程或非阻塞 I/O 来防止整个程序被卡住。

**从 CS111 角度看**：在操作系统实现中，当用户进程发出一个慢速系统调用时，内核会将该进程置于睡眠状态（`TASK_INTERRUPTIBLE` 或 `TASK_UNINTERRUPTIBLE`），并将 CPU 调度给其他就绪的进程。这就是为什么一个阻塞的系统调用不会浪费 CPU——被阻塞的线程根本不消耗 CPU 时间片。
:::

# 使慢速系统调用变快
* 将描述符配置为非阻塞。
    * 可以将描述符配置为所谓的**非阻塞**。当非阻塞
      描述符传递给 `accept`、`read` 或 `write` 时，
      函数将始终尽可能快地返回，而不等待任何外部事物。
    * `accept`
        * 如果在调用 `accept` 时有客户端连接可用，它将立即返回一个
          到该客户端的 socket 连接。
        * 如果服务器 socket 被配置为非阻塞，当没有待处理的连接请求时，
          `accept` 将返回 -1 而不是阻塞。这个 -1 通常表示发生了某个错误，
          但如果全局变量 `errno` 被设置为 `EWOULDBLOCK`，
          那么这个 -1 并不是***真正***标识一个错误，而是说如果传递给它的服务器 socket
          是传统的（即阻塞的）socket 描述符，`accept` 就会阻塞。
    * `read`
        * 如果在调用 `read` 时有一个或多个字节的数据立即可用，那么
          这些字节（或至少其中一些）将被写入提供的字符缓冲区，并返回
          已放置的字节数。
        * 如果没有数据可用（且描述符仍然是打开的，并且描述符的另一端尚未关闭），
          那么 `read` 将返回 -1，前提是该描述符已被配置为非阻塞。
          同样，这个 -1 通常表示发生了某个错误，但在这种情况下，
          全局变量 `errno` 被设置为 `EWOULDBLOCK`。这是我们用来判断
          `read` 并没有***真正***失败的线索。-1/`EWOULDBLOCK` 组合只是说明
          如果描述符是传统的（即阻塞的）描述符，对 `read` 的调用***本来会***阻塞。

::: tip 重难点解析
**非阻塞 I/O 的返回值语义**：在阻塞模式下，`read` 返回 -1 只意味着一件事：出错了。但在非阻塞模式下，`read` 返回 -1 可能有两种含义：(1) 真的出错了；(2) 当前没有数据可读，但以后可能有。区分这两种情况的唯一方法就是检查 `errno`——如果 `errno == EWOULDBLOCK`（或 `EAGAIN`，两者在 Linux 下等价），则说明"暂时无数据，稍后再试"。这种"返回错误码来表达非错误状态"的设计，是 UNIX 中非阻塞 I/O 编程需要特别留意的惯用法。

**非阻塞 I/O 并不能让慢速调用变快**：所谓"使慢速系统调用变快"实际上是一个误导性的表述。非阻塞 I/O 并没有加速 I/O 本身——数据的到达速度完全取决于网络和硬件。它的真正价值在于允许调用线程不被阻塞，从而在一个线程内管理多个连接。这就是事件驱动编程（event-driven programming）和 `epoll`/`kqueue` 等 I/O 多路复用技术的基石。
:::

::: tip 重难点解析
**抽象 —— 计算机科学中最重要的思想**：抽象的本质是"关注接口而非实现"。`sort` 函数是一个完美的抽象示例：你只需要知道它能排序，而不需要知道它底层用的是快速排序、归并排序还是堆排序。在 CS110 中，文件描述符是将"一切皆文件"的 UNIX 哲学变为现实的核心抽象——无论是磁盘文件、终端、管道还是网络 socket，你都可以用 `read` 和 `write` 来操作，因为这些操作符的接口是一致的。
:::

::: tip 重难点解析
**抽象的反面 —— 泄密的抽象 (Leaky Abstractions)**

Joel Spolsky 在 2002 年提出了"泄密抽象法则"（Law of Leaky Abstractions）：所有非平凡抽象在某种程度上都会泄密——即当底层实现细节突破抽象层，迫使用户必须理解底层才能正确使用。

**CS110 中的实例**：

1. **文件描述符抽象并非总是完美**：`read`/`write` 对 socket 的表现与对本地文件不同——socket 可能分片到达数据（TCP 是字节流，没有消息边界），而文件总是完整可读。如果假设"一次 `read` 就能拿到完整的 HTTP 请求"，在 socket 编程中就会出 bug。

2. **线程抽象泄露了竞争条件**：`thread` 对象让你以为创建并发执行就像创建一个对象那么简单，但内存模型的细节（缓存一致性、指令重排、原子操作的必要性）会泄露出来——你必须用 `mutex`、`atomic` 等工具来修补。

3. **虚拟内存抽象泄露了性能边界**：每个进程看起来都拥有完整的地址空间，但 TLB 缺失和页错误（page fault）会让你感受到物理内存和页表的真实存在。在 CS111 中你会用 `perf stat -e page-faults` 看到这个抽象的花费。

**关键教训**：理解抽象下面的层不是"额外的工作"——它是成为高效系统程序员的必要条件。抽象帮你入门并提高生产力，但当性能瓶颈出现或异常发生时，你必须能够"向下看"一层。
:::

::: tip 重难点解析
**模块化与分层 —— 构建大型系统的核心方法论**：模块化的本质是"分而治之"——将一个复杂的大问题分解为多个明确的小模块，每个模块有清晰的接口。分层是模块化的一种特殊形式，各层之间有严格的上下依赖关系。这一思想从 CS110 的文件系统、编译器到 Facebook 这样的超级系统，贯穿始终。关键原则：每一层只向上暴露接口、只向下调用服务。这种单向依赖使得每一层可以被独立替换或优化，而不会影响整个系统。

**现实系统中的分层不是"完美"的**：从 Facebook 的例子可以看到，在真实的巨型系统中，并非所有模块都严格遵守严格分层。有些层（如缓存）可能被多个上层共享，有些子系统可能跨越多个层。工程上的分层是实用的设计指导原则，而非不可变通的教条。
:::

::: tip 重难点解析
**端到端原则 (End-to-End Principle) —— 分层不是万能的**

Saltzer, Reed 和 Clark 在 1984 年的经典论文中提出了"端到端原则"（end-to-end argument）：通信系统中的许多功能（如可靠传输、加密、错误检测）只有放在通信端点才能真正保证正确性，放到中间层次（如网络层）只能提高性能，不能替代端点检查。

**经典例子：文件传输中的校验**

```
App A                                      App B
  |--- checksum over file data --->         |
  |                                         |
[TCP] --> [IP] --> [ethernet] --> ...    [TCP]
```

TCP 提供校验和来检测传输中的比特错误，但：
- 磁盘在写入文件前可能有内部控制器的缓冲区错误
- 应用程序的内存中可能发生比特翻转（bit flip）
- 中间路由器可能在 TCP 校验和计算后引入错误

因此，正确做法是在应用层对整个文件计算强校验和（如 SHA-256），而 TCP 校验和只是一个性能优化——帮助较早发现问题，但不能替代端到端的完整性验证。

**CS110 中的体现**：
- HTTP 协议在 TCP 之上（应用层），但它仍需处理错误状态码（4xx, 5xx）和部分内容——因为 TCP 只保证字节正确传输，不保证业务语义正确
- DNS 解析虽然依赖递归服务器，但本地存根解析器（stub resolver）仍然可以做最终验证
- 作业 2 的文件系统：路径名层依赖 inode 层，但路径名层仍然需要处理自己的错误（如路径不存在、权限不足）

**模块化的度量 —— 内聚与耦合 (Cohesion and Coupling)**

- **高内聚**：模块内部的元素密切相关，服务于同一目的。反例：一个既处理文件 I/O 又管理 GUI 的巨型类。
- **低耦合**：模块之间的依赖尽可能少、尽可能弱。测量方法——**关联度 (connascence)**：若 A 的修改强制 B 也必须修改，则 A 和 B 具有关联度。关联度可按"距离"分类，如名称关联度（connascence of name, 最低）到意义关联度（connascence of meaning, 最高）。好设计追求最低的关联度类型。
:::

              我们在课堂上花时间深入理解了人类可读的域名是如何
              转换为 IP 地址的。
        * 其他示例：URL（资源位置的人类可读形式）、
          进程 ID（引用进程的计算机友好方式）、
          以及文件描述符（引用文件[或行为类似文件的东西——虚拟化和抽象万岁！]的计算机友好方式）
    * 缓存（Caching）
    * 虚拟化（Virtualization）
    * 并发（Concurrency）
    * 客户端-服务器请求与响应（Client-server request-and-response）

::: tip 重难点解析
**命名与名称解析 —— 系统中最常见的间接层**：几乎所有计算机系统问题都可以通过增加一层间接来解决。命名和名称解析正是这一思想的具体体现。路径名 -> inode 号、域名 -> IP 地址、进程 ID -> 进程控制块（PCB）、文件描述符 -> 打开文件表项——这些都是将人类友好的（或用户空间友好的）名称映射到底层实现的经典案例。名称解析的关键挑战在于性能（目录查找可能涉及多次磁盘 I/O、DNS 解析可能需要网络往返）和一致性（名称绑定是否保证最新）。

**从 CS111 的虚拟内存角度看**：虚拟地址到物理地址的转换也是一种名称解析——页表（page table）和 TLB 就是这一解析过程的核心数据结构。CS110 中的文件系统名称解析（路径 -> inode -> 块）在逻辑上与 CS111 中的地址解析（虚拟地址 -> 物理地址）高度相似，都经历了多级查表的过程。
:::

::: tip 重难点解析
**命名解析的经典权衡 —— DNS vs /etc/hosts**

这两种名称解析机制展示了分布式 vs 本地化命名的根本权衡：

| 特性 | DNS | /etc/hosts |
|------|-----|------------|
| 可发现性 | 高（全球命名空间，人人可查询） | 低（仅本机可见） |
| 本地控制 | 低（需要 DNS 管理员权限） | 高（root 可随时修改） |
| 延迟 | 变数（需网络往返，可能多次递归查询） | 极低（直接文件读取） |
| 一致性 | 最终一致性（有 TTL，变更传播慢） | 即时（修改立即生效） |
| 可扩展性 | 极高（分层委托，百万级域） | 极低（纯文本文件，适合少量条目） |
| 故障模式 | 网络分区时无法解析 | 只要文件系统正常即可工作 |

**CS110 中的类似权衡**：
- 作业 2 的文件系统：inode 号（机器友好）vs 路径名（人类友好）
- 网络编程中：`gethostbyname`（依赖 DNS）vs 硬编码 IP 地址
- 进程管理：PID（由内核分配，紧凑快速）vs 进程名（人类可读，但可能重复）

在 CS111 中，你还会看到虚拟地址空间的命名：虚拟地址（进程视角的线性空间）vs 物理页框号（硬件实际存储位置）。页表本质上是 process-local 的"命名空间映射表"。
:::

# 系统设计原则
* CS110 涉及七个这样的原则：
    * 抽象（Abstraction）
    * 模块化和分层（Modularity and Layering）
    * 命名和名称解析（Naming and Name Resolution）
    * 缓存（Caching）
        * 简单来说，缓存是一个组件——可能在硬件中，也可能在软件中——
          它存储数据以便未来的请求可以更快地被处理。
        * 基于地址的缓存的经典示例（如 CS107 中所教授）
            * L1 级指令和数据缓存，用作 CPU 寄存器的中转区域（staging area）。
            * L2 级缓存，用作 L1 缓存的中转区域。
            * 主内存的一部分——不用于支持活动进程虚拟地址空间的那部分——
              用作磁盘缓存来存储文件页面。
        * 存储重复（通常是昂贵的）计算结果缓存的示例：
            * Web 浏览器缓存最近获取的文档，前提是服务器明确表示
              这些文档可以被缓存，无论是无限期还是在一定时间内。
            * Web 代理缓存静态资源，以便*其他*请求相同数据的客户端
              可以更快地得到服务。
            * DNS 缓存，保存最近解析的域名到其 IP 地址的映射。
            * [memcached](http://www.memcached.org)，维护一个经常用于
              生成 Web 内容的对象字典。
            * [Solaris 文件系统层](http://www.princeton.edu/~unix/Solaris/troubleshoot/diskio.html#fscache)中的四种不同缓存
                * 目录名查找缓存（Directory Name Lookup Cache）存储 vnode 条目到路径目录查找信息。
                * inode 缓存收集并维护更永久地存储在 inode 中的文件元数据信息。
                * rnode 缓存存储关于构成 NFS 文件系统的许多节点的信息。
                * 缓冲区缓存（buffer cache）存储 inode 和间接磁盘块。
    * 虚拟化（Virtualization）
    * 并发（Concurrency）
    * 客户端-服务器请求与响应（Client-server request-and-response）

::: tip 重难点解析
**缓存无处不在 —— 计算机系统中最重要的性能优化手段**：缓存的思想基于一个简单的观察——访问快速存储比访问慢速存储快得多（寄存器 > L1 缓存 > L2 缓存 > 主存 > SSD > 磁盘 > 网络），而程序往往表现出"时间局部性"（最近访问的数据很可能再次被访问）和"空间局部性"（附近的数据很可能被访问）。缓存在 CS110 的各个作业中都有体现：作业 2 的文件系统块缓存、作业 7 的 HTTP 代理缓存、DNS 结果的本地缓存等。

**从 CS111 的角度看 TLB 缓存**：TLB（Translation Lookaside Buffer）是虚拟地址到物理地址转换的专用硬件缓存，它是最讲究性能的缓存之一——TLB 缺失可能导致多次内存访问，开销极大。这解释了为什么操作系统使用大页（huge pages）来减少 TLB 条目的使用量。
:::

::: tip 重难点解析
**缓存一致性问题 (Cache Coherence) —— 缓存的核心难点**

缓存带来的最大挑战不是"怎么存"，而是"何时失效"（invalidation）。这是缓存设计的核心问题：

**问题**：当原始数据被修改后，缓存中的副本变得陈旧（stale）。如何确保客户端不会消费到过期数据？

**两种主要的失效策略**：

1. **TTL-based（基于过期时间）**：
   - 每个缓存条目分配一个存活时间（Time-To-Live），过期后自动失效
   - 优点：简单，无需协调；缺点：在 TTL 窗口内可能返回过期数据
   - 实例：DNS 缓存（TTL 由域名的权威服务器设定）、HTTP `Cache-Control: max-age=3600`
   - CS110 中：你的 HTTP 代理可以基于 `Cache-Control` 头管理缓存条目生命周期

2. **Event-driven（基于事件驱动）**：
   - 当原始数据修改时，主动通知所有缓存副本使其失效
   - 优点：缓存始终一致（无过期窗口）；缺点：需要维护失效通知的基础设施
   - 实例：CPU cache coherence protocol (MESI 协议)、分布式缓存系统写失效 (write-invalidate)
   - 挑战：分布式场景下通知可能丢失或延迟，需要额外的确认和重试机制

**CS110 作业 7 (HTTP 代理) 中的缓存设计**：
- 简单方案：用 LRU 驱逐 + 固定 TTL（如 30 秒），适合教学但不够精确
- 进阶方案：解析 `Cache-Control` / `ETag` 头，实现条件请求 (`If-None-Match`) 来验证缓存有效性——这是 HTTP 协议中"缓存一致性"的标准方式

**从 CS111 视角看 CPU 缓存一致性**：多核 CPU 中每个核心有自己的 L1/L2 缓存。当 Core A 修改一个内存位置时，Core B 的缓存中该位置的副本必须被标记为无效。硬件通过 **MESI 协议**（Modified/Exclusive/Shared/Invalid）在总线/Bus 层面自动处理这一问题。理解硬件级别的缓存一致性是理解多线程程序中 `volatile` 和 `atomic` 关键字底层含义的必要前提。
:::

# 系统设计原则
* CS110 涉及七个这样的原则：
    * 抽象（Abstraction）
    * 模块化和分层（Modularity and Layering）
    * 命名和名称解析（Naming and Name Resolution）
    * 缓存（Caching）
    * 虚拟化（Virtualization）
        * 虚拟化是一种抽象机制，用于使多个硬件资源
          看起来像一个。示例包括：
            * RAID，将多个（通常廉价的）存储设备聚合
              为一个独立的硬盘驱动器。
            * **Andrew 文件系统**（Andrew File System，简称 AFS），将许多
              独立的、联网的文件系统嫁接成一个以 `/afs`
              为根的单一系统。你进入 `/afs` 的各个子目录时，
              原则上不需要知道这些目录在世界上的哪个地方存储。
            * Web 服务器负载均衡器，将成百上千甚至
              数十万台服务器的前端放置一小部分机器集合，
              拦截所有请求并将其快速转发到负载最少的服务器。
        * 虚拟化*也*是一种抽象机制，用于使单个硬件资源
          看起来像多个。示例包括：
            * 虚拟到物理内存映射，允许每个进程
              相信它拥有整个地址空间。我们几周前在课堂上讲过虚拟到物理
              内存映射。
            * 线程，单个进程的栈帧被细分为许多栈帧，
              以便多个执行线程，每个线程都有自己的上下文，
              可以以与调度器轮转多个进程大致相同的方式进行轮转。（线程实际上常常
              被称为**虚拟进程**（virtual processes）。一个进程被做成看起来像多个）。
            * 虚拟机，它们是设计用来像物理机器一样执行程序的软件实现。
              虚拟机可以做很少的事情，比如为 Java 或 C# 可执行文件提供运行时环境，
              也可以做很多的事情，比如在一个原本不支持多个操作系统的架构上运行
              多个不同的操作系统（Mendel Rosenblum 的 VMware 就是一个例子）。
    * 并发（Concurrency）
    * 客户端-服务器请求与响应（Client-server request-and-response）

::: tip 重难点解析
**虚拟化的两个方向 —— 聚合与切分**：虚拟化的精髓在于"欺骗"——让上层认为下层的资源形态与实际情况不同。它有两个对偶方向：(1) 多对一聚合：RAID 把多个磁盘"假装"成一个、负载均衡器把多台服务器"假装"成一台；(2) 一对多切分：虚拟内存让每个进程以为自己独占全部地址空间、虚拟机让一台物理机上运行多个操作系统。这两种方向都服务于同一个目的——隔离复杂性，为上层提供更简单、更灵活的接口。

**关于线程即"虚拟进程"**：这个比喻非常精准。在早期的 UNIX 中只有进程，后来人们发现进程创建和切换开销太大，于是发明了线程。从内核调度器的角度来看，它调度的最小单位是线程（而非进程）——所以在 Linux 中线程和进程使用相同的数据结构（`task_struct`），唯一的区别在于是否共享地址空间。CS111 中会详细讲解线程的 `clone()` 系统调用实现，你会发现它和 `fork()` 共用大量内核代码。
:::

::: tip 重难点解析
**虚拟化的代价 —— 具体的性能开销数字**

虚拟化让系统更灵活，但不是免费的。以下是关键操作的数量级开销（以现代 x86 服务器为例，约 2020 年代硬件）：

| 操作 | 典型耗时 | 备注 |
|------|---------|------|
| CPU 寄存器访问 | ~0.3 ns | 一级存储，零开销 |
| L1 缓存命中 | ~1 ns | 40-50 周期 @ 4 GHz |
| L2 缓存命中 | ~4 ns | |
| L3 缓存命中 | ~12 ns | |
| 主存访问 | ~100 ns | 约 400 周期！ |
| 系统调用（空 `getpid`） | ~150 ns | 用户态/内核态切换 |
| 上下文切换（同进程线程） | ~1-2 us | 保存/恢复寄存器 + FPU/SSE 状态 |
| 上下文切换（不同进程） | ~3-5 us | 加上 TLB flush + 页表切换 |
| `fork()` + `exec()` | ~400 us | 复制页表条目 |
| SSD 随机读取 (4KB) | ~100 us | 约等于 400,000 次 L1 缓存访问 |
| 磁盘寻道 | ~5-10 ms | 约等于 2-5 千万次 CPU 周期 |

**关键洞察**：
- 一次磁盘寻道的时间可以执行约 100 万次系统调用，或 5000 万条 CPU 指令
- `fork()` 的写时复制（COW）机制让你"感觉" fork 很快（页表复制只需 ~0.4ms），但真正的开销延迟到了子进程写入已复制页面时——第一次写入触发 page fault，内核需要实际分配和复制物理页框（额外 ~5us/页）
- 虚拟内存的 TLB 映射是"无感"的（在 L1 缓存速度级别），但当 TLB 缺失时（每次上下文切换后必然发生前几次访问），每次访问要经历 4 级页表遍历（约 4 x 100ns = 400ns），这就是上下文切换的隐性成本

**CS110 中的体现**：为什么非阻塞 I/O + epoll 的单线程模型能比多线程模型在大量连接时表现更好？因为每个线程都需要上下文切换（~2us），当有 10000 个线程时，调度器的时间片轮转本身就有显著开销，而单线程事件循环完全消除了上下文切换。
:::

* CS110 涉及七个这样的原则：
    * 抽象（Abstraction）
    * 模块化和分层（Modularity and Layering）
    * 命名和名称解析（Naming and Name Resolution）
    * 缓存（Caching）
    * 虚拟化（Virtualization）
    * 并发（Concurrency）
        * 我们在并发方面已经有很多经验了：
            * 多个进程在单个处理器上运行，看似同时进行。
            * 多个线程在单个进程内运行，看似同时进行。
        * 当有多个处理器和/或多个核心可用时，进程可以真正
          并行运行，单个进程内的线程也可以并行运行。
        * 信号和中断处理程序在技术上也是并发程序。程序
          执行偶尔需要暂停以接收来自外部（文件系统、键盘、鼠标、
          Wacom 平板、连接的 MIDI 控制器或网络）的信息。
        * 一些编程语言——[Erlang](http://www.erlang.org/) 就是一个
          例子——天生就是并发的，以至于它们采用了一种使竞争条件几乎不可能的编程模型。
          （其他语言——我想到了纯 JavaScript——则采用并发或至少是线程化太复杂且容易出错而不值得支持的观点）。
        * 到了周三，你将沉浸在 MapReduce 作为编程模型的世界中，
          该模型使用跨多个机器的多个处理器的多个线程或进程来处理
          大量信息，并将这些信息削减成有用的、聚合的数据集。
    * 客户端-服务器请求与响应（Client-server request-and-response）

::: tip 重难点解析
**并发（Concurrency）vs 并行（Parallelism）**：这两个概念经常被混用，但它们的含义不同。并发是逻辑上的同时执行——多个任务可以在同一时间段内推进，但不一定在同一时刻执行（单核 CPU 通过时间片轮转即可实现并发）。并行是物理上的同时执行——多个任务在多个处理器核心上同时刻运行。CS110 中你遇到的竞争条件和互斥锁问题来自并发，而用 MapReduce 加速大规模数据处理则利用了并行。

**不同语言对并发的态度**：Erlang 采用 Actor 模型，每个 Actor 之间通过消息传递通信，天然避免了共享内存带来的竞争条件。Node.js 通过事件循环 + 回调的单线程模型实现高并发，避免了多线程的复杂性。这些不同的设计选择反映了"并发"并非只有一种范式——选择哪种取决于你的场景、语言和偏好。
:::

::: tip 重难点解析
**Amdahl 定律 —— 并行加速的数学上限**

Gene Amdahl 在 1967 年提出了一个简单却深刻的公式，它定义了并行化加速的理论极限：

```
Speedup = 1 / (S + (1-S)/N)
```

其中：
- **S** = 程序中必须串行执行的部分所占比例（0 <= S <= 1）
- **N** = 处理器数量

**关键推论**：

| 串行比例 S | N=4 | N=16 | N=64 | N=∞ (极限) |
|-----------|-----|------|------|-----------|
| 1% | 3.88x | 13.9x | 39.3x | 100x |
| 5% | 3.48x | 9.1x | 16.4x | 20x |
| 10% | 3.08x | 6.4x | 9.1x | 10x |
| 25% | 2.29x | 3.4x | 3.7x | 4x |
| 50% | 1.60x | 1.9x | 2.0x | 2x |

**震撼的结论**：如果程序的 10% 是串行的（S=0.1），无论你有多少处理器，加速比永远无法超过 10 倍。这就是为什么在实际系统中，仅仅增加 CPU 核心不能无限提升性能——你最终会被程序的串行部分所限制。

**CS110 中的应用**：

1. **MapReduce**：理想情况下 map 和 reduce 阶段都是高度并行的（S 很小），但 shuffle 阶段（数据通过网络传输和排序）包含串行瓶颈——所有中间数据必须通过网络传输完毕才能开始 reduce。如果你的 MapReduce 作业的 shuffle 占总时间的 20%，那么 S >= 0.2，你最多只能获得 5 倍加速。

2. **多线程服务器**：`accept` 和 `ThreadPool.schedule` 是串行操作——主线程必须串行接受连接然后分派。当连接速率极高（如 100K 连接/秒），`accept` 本身成为瓶颈。

3. **锁竞争**：如果多个线程频繁争用同一把 `mutex`（如全局日志锁），这个被锁保护的代码段实际上是串行的——这增加了程序的 S，降低了
可扩展性。这就是为什么细粒度锁（fine-grained locking）和无锁数据结构（lock-free data structures）在高并发场景中如此重要。

**Gustafson 定律的补充**：Amdahl 定律假设问题规模固定。Gustafson 在 1988 年指出，在实践中我们通常不会用更多处理器来解决相同规模的问题——我们会用更多处理器来解决更大规模的问题。在这种情况下，串行部分的比例可能因问题规模增长而下降。这就是为什么 MapReduce 可以在数千台机器上有效运行——因为数据规模随处理器数量增长而增长。
:::

# 系统设计原则
* CS110 涉及七个这样的原则：
    * 抽象（Abstraction）
    * 模块化和分层（Modularity and Layering）
    * 命名和名称解析（Naming and Name Resolution）
    * 缓存（Caching）
    * 虚拟化（Virtualization）
    * 并发（Concurrency）
    * 客户端-服务器请求与响应（Client-server request-and-response）
        * 请求/响应（Request/response）是一种将功能组织成具有明确
          职责集合的模块的方式。
        * 我们已经有了一些关于请求与响应方面的经验。
            * 系统调用（`open`、`write`、`fork`、
              `sleep`、`bind` 等）都是用户空间
              围绕一种特殊类型的内核调用的包装器……用户空间和内核空间
              是两个独立的模块，它们之间有非常非常明确的边界）。
            * HTTP、IMAP
            * NFS、AFS
            * DNS

::: tip 重难点解析
**客户端-服务器模型 —— 系统设计的"元模式"**：这第七个原则实际上是一种组织系统交互的通用范式。系统调用是用户程序（客户端）向内核（服务器）发起请求；HTTP 是浏览器（客户端）向 Web 服务器发起请求；DNS 是应用程序（客户端）向域名服务器发起请求。请求-响应模式的本质是：一方主动发起请求，另一方被动响应。这种非对称性简化了系统的设计，因为职责划分非常清晰——客户端发起、服务器处理和回复。

**在 CS110 中，客户端-服务器模型贯穿始终**：从 `open`/`read`/`write`（进程请求内核服务），到 `fork`/`execvp`（进程请求内核创建新进程），到 socket 编程（客户端请求服务器资源），甚至到 MapReduce（mapper/reducer 之间通过文件系统进行"请求-响应"式的数据交换）——这门课程事实上一直在教你构建和调试各种形式的客户端-服务器交互。
:::
