::: tip 重难点解析
**Unix v6 文件系统的 inode 结构**：这是理解文件系统实现的经典教材。每个文件在磁盘上对应一个唯一的 inode（索引节点），inode 内部包含文件的元数据（权限、大小、时间戳等）和一个多层级的块指针数组。具体来说，inode 中的指针结构包含：

- **直接块（Direct Blocks）**：前若干个指针（例如前 8 个）直接指向存储文件数据的磁盘块。对于小文件（如 18 字节的文件），一个直接块就足够容纳全部数据。
- **间接块（Indirect Blocks）**：当文件超过直接块能寻址的大小时，下一个指针指向一个间接块，该块中存储的不是文件数据，而是更多指向数据块的指针。这大大扩展了寻址空间。
- **双重间接块（Double-Indirect Blocks）**：对于更大文件，inode 的最后几个指针指向双重间接块——该块中的每个指针指向一个间接块，每个间接块再指向数据块。这种两级索引结构使得一个 inode 可以表达任意大的文件。

**为什么用这种分级结构？** 大多数文件非常小（想想源代码文件、配置文件），直接块能覆盖绝大多数场景，保证小文件的访问速度。而间接块机制确保了文件系统也能支持超大文件，尽管访问这些文件尾部时需要额外的磁盘读取来解析间接块。这是一种典型的"常见情况快速、罕见情况可行"的系统设计权衡。CS111 课程中虚拟内存的"多级页表"也采用了完全相同的设计理念。
:::

::: tip 重难点解析
**inode 指针结构的精确计算——从字节到 GB 的跨越**：

Unix v6 文件系统（以及现代 ext2/ext3/ext4）中，每个 inode 包含一个固定大小的指针数组。假设块大小 = 4KB（4096 字节），每个指针 4 字节（32 位块编号），则每个块可存储 1024 个指针。

**标准 inode 布局（以 ext2/ext3 为例）**：

```
直接块 (direct blocks):       条目 [0]..[11]  —— 12 个直接指针
单级间接 (single indirect):   条目 [12]        —— 指向一个装满指针的块
双级间接 (double indirect):   条目 [13]        —— 指向一个装满指针的块，其中每个指针再指向一个装满指针的块
三级间接 (triple indirect):   条目 [14]        —— 三层指针块嵌套
```

**可达文件大小计算**：

```c
#define BLOCK_SIZE  4096          // 4KB
#define PTRS_PER_BLOCK  (BLOCK_SIZE / 4)  // 1024 个指针/块

// 直接块：12 个直接指针
direct_max = 12 * 4096 = 48 KB

// 单级间接：1 个间接块 → 1024 个数据块
single_indirect_max = 1024 * 4096 = 4 MB

// 双级间接：1 个二级块 → 1024 个一级块 → 1024*1024 个数据块
double_indirect_max = 1024 * 1024 * 4096 = 4 GB

// 三级间接：1 个三级块 → 1024*1024*1024 个数据块
triple_indirect_max = 1024 * 1024 * 1024 * 4096 = 4 TB
```

**为什么是 12 个直接块？** 大多数文件小于 48KB——源代码、配置文件、日志条目。这一设计保证了对这些小文件的访问只需一次磁盘读取获取 inode（加上读取数据块的 I/O），无需解析间接块。对于大文件，额外的间接块带来的开销与数据量相比可忽略。

**与虚拟内存页表的对比**：文件系统 inode 的多级索引和虚拟内存页表的多级页表是同构的设计——都是"用固定大小的顶层结构 + 分级指针扩展寻址范围"。为什么不用线性数组？因为大多数 inode/进程的地址空间是稀疏的（小文件/小进程居多），线性数组会浪费大量空间。
:::

::: tip 重难点解析
**命名与名称解析——系统设计的统一原理**：文件系统和 DNS 的相似性并非巧合，它们都体现了"命名与名称解析"这一核心系统设计原则。两种系统都采用了分层命名空间（hierarchical namespace）——文件系统用 `/` 分隔的路径（如 `/usr/bin/gcc`），DNS 用 `.` 分隔的域名（如 `www.stanford.edu`）。两者的解析过程也都是自顶向下的：文件系统从根目录 `/` 开始逐级查找 inode，DNS 从根域名服务器开始逐级查找权威服务器。

**分层命名的好处**：
1. **去中心化管理**：每个中间节点（目录/域）可以独立管理其下的命名空间（如 `cs110/` 目录由课程管理，`stanford.edu` 域由斯坦福管理）。
2. **可扩展性**：理论上无限的命名空间，实际实现受限于 inode 指针结构（文件系统）或查询超时（DNS）。
3. **容错性**：某个子树失效不影响其他部分。

这两种系统的对比学习可以帮助你建立跨领域的系统设计直觉。CS111 课程在讨论文件系统时会进一步深化这些概念。
:::

    * Unix v6 文件系统设计和 DNS 架构都依赖于命名和名称解析这一事实，说明了人们普遍认为的**计算机系统中的命名**是**系统设计的基本原理**。

::: tip 重难点解析
**计算机系统中的命名**：Saltzer 和 Kaashoek 的教材将"命名"（naming）列为计算机系统设计的核心原则之一。命名的本质是在"人类可理解的名字"和"机器可处理的标识符"之间建立映射关系。在整个 CS110 课程中，你会反复看到命名的应用：

- **文件系统**：文件名 -> inode 编号 -> 磁盘块编号
- **DNS**：域名 -> IP 地址
- **虚拟内存**：虚拟地址 -> 物理地址（通过页表）
- **网络**：主机名 -> IP 地址（通过 DNS 或 `/etc/hosts`）
- **RPC/REST**：服务名 -> 网络端点

理解这一原理将帮助你从更高的抽象层次看待不同的计算机系统模块。
:::

::: tip 重难点解析
**DNS 解析的完整递归过程——从根到叶的逐级查询**：

DNS 解析采用分层递归（或迭代）查询。以解析 `www.stanford.edu` 的 A 记录为例：

```
客户端                                                  DNS 服务器
  |                                                        |
  |  1. 查询 www.stanford.edu                               |
  |------------------------------------------------------->|
  |                                                        |  2. 问根服务器: who handles .edu?
  |                                                        |--------> [a.root-servers.net]
  |                                                        |<-------- edu 的 NS 记录 + glue records
  |                                                        |
  |                                                        |  3. 问 .edu TLD: who handles stanford.edu?
  |                                                        |--------> [a.edu-servers.net]
  |                                                        |<-------- stanford.edu 的 NS 记录
  |                                                        |
  |                                                        |  4. 问 stanford.edu 权威: what is www?
  |                                                        |--------> [ns1.stanford.edu]
  |                                                        |<-------- A 记录: 171.67.215.200
  |                                                        |
  |  5. 返回 171.67.215.200                                 |
  |<-------------------------------------------------------|
```

**Glue Records（粘合记录）——解决"先有鸡还是先有蛋"**：当权威 DNS 服务器位于自己管理的域内时会出现悖论。例如 `stanford.edu` 的权威 DNS 是 `ns1.stanford.edu`——要解析 `ns1.stanford.edu` 的 IP，需要先问 `ns1.stanford.edu`，这就形成了循环依赖。Glue records 在上级域（`.edu` TLD）中提供了额外的 A 记录，直接给出 `ns1.stanford.edu` 的 IP 地址，打破了这个循环。

**为什么是 13 个根服务器？** 这不是随意定的数字。DNS 响应最初通过 UDP 传输，早期标准要求 DNS 响应不得超过 512 字节（UDP 在不分片情况下的安全大小）。一个响应包需要包含 13 个根服务器的名称和 IPv4 地址（每个约 16 字节名称 + 4 字节 IP = 20 字节，加上开销），512 字节恰好能容纳 13 个根服务器的信息。这 13 个"逻辑"根服务器（a.root-servers.net 到 m.root-servers.net）背后，通过 **anycast（任播）** 技术在全球部署了数百个物理实例——多个物理服务器共享同一 IP 地址，BGP 路由协议将用户请求自动导向最近的实例，实现了负载均衡和低延迟。

**递归 vs 迭代查询**：
- **递归查询**：DNS 客户端要求服务器替自己完成全部解析工作（服务器承担工作量）。大多数 ISP 的 DNS resolver 接受递归查询。
- **迭代查询**：DNS 服务器返回"下一步该问谁"的线索（客户端承担工作量）。根服务器和 TLD 服务器通常只接受迭代查询，以减轻负载。

**`gethostbyname` vs `getaddrinfo`**：CS110 中使用的是传统的 `gethostbyname`（仅支持 IPv4，线程不安全，已过时），现代代码应使用 `getaddrinfo`（支持 IPv4/IPv6，线程安全，支持协议和套接字类型过滤）。
:::
