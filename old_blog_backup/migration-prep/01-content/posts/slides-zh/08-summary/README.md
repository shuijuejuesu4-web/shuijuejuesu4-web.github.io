::: tip 重难点解析
**虚拟地址空间的"错觉"**：每个进程都"以为自己"独占整个地址空间（32 位系统下是 4 GB，64 位系统下理论上是 256 TB），但实际上物理内存要小得多。这是通过虚拟内存（virtual memory）机制实现的——硬件 MMU（内存管理单元）负责将每个进程的虚拟地址翻译为物理地址。这个"幻觉"是多进程的基础：各进程相互隔离，一个进程的崩溃不会直接影响到其他进程。在 CS111 课程中，你会深入学习分页（paging）、页表（page table）和 TLB（Translation Lookaside Buffer）的具体实现。

**调度器（Scheduler）的角色**：当系统中存在多个就绪进程时，调度器决定"CPU 当前应该运行哪个进程"。轮转调度（round-robin）是最基本的策略——每个进程依次运行一小段时间片（time slice），超时后切换。这保证了公平性，但可能不是最优的（实时系统、交互式系统有不同的需求）。CS111 会深入讨论调度算法的设计权衡，包括多级反馈队列（MLFQ）、完全公平调度器（CFS）等。
:::

# 虚拟地址空间

* 该图为一个运行在 32 位地址空间的现代 Linux 系统的典型虚拟地址空间示意图（来自 B&O 教材，带有额外的语义标注）。

  * 为了让进程产生其运行在所有 4 GB（或 232 字节）能力之上的错觉，机器以虚拟地址（也称为逻辑地址）运行。
  * 数据和指令对应于物理随机存取存储器（random-access memory，RAM）本身的虚拟地址可以存在，如果被请求的数据不在 RAM 中，则会触发内核从交换空间中将它们交换回来。
  * 高 896 MB 部分称为内核空间，它与所有进程共享，并且只能由执行在内核模式下的指令访问。内核级栈也在高地址（高虚拟页号）范围内。
  * 低 3 GB 部分称为用户空间，由用户级栈（从高地址向低地址增长）、动态链接库（通常称为共享对象）、堆（heap，从其他段向栈方向增长）、BSS 段、数据段和文本段组成。

  ```
                             +-------------------+
                        4GB  |                   |
                             |                   |
                             |                   |
                             |     Kernel        |
                             | (代码和数据)       |
                             +-------------------+
                        3GB  |                   |
                             |  用户级栈 (Stack)  |
                             |      (⬇)          |
                             :         :         :
                             :         :         :
                             |      (⬆)          |
                             |   (内存映射段)     |
                             | 共享库 (Shared Libs) |
                             +-------------------+
                             |      堆 (Heap)     |
                             |                   |
                             +-------------------+
                             |     BSS 段        |
                             +-------------------+
                             |     数据段         |
                             +-------------------+
                             |     文本段         |
                             +-------------------+
                        0GB  |                   |
                             +-------------------+
  ```

::: tip 重难点解析
**用户空间各段的含义**：

- **文本段（Text Segment）**：存放程序的机器指令，通常只读。
- **数据段（Data Segment）**：存放已初始化的全局变量和静态变量。
- **BSS 段（Block Started by Symbol）**：存放未初始化的全局变量和静态变量（在可执行文件中不占空间，运行时由 OS 清零分配）。
- **堆（Heap）**：由 `malloc`/`free` 管理的动态内存区域，从低地址向高地址增长。
- **共享库（Shared Libraries）**：如 `libc.so`，映射到用户地址空间的中间区域，供多个进程共享。
- **用户级栈（User Stack）**：存放函数调用帧、局部变量，从高地址向低地址增长。栈和堆相向增长的设计最大化了可用空间。

**内核空间**：虚拟地址空间顶部（3 GB 以上，32 位系统）保留给内核。这部分地址在所有进程的地址空间中映射到相同的物理内存，但只有 CPU 在内核模式下才能访问。系统调用、中断处理都在内核空间执行。这种"分割"设计实现了用户态与内核态的隔离——用户程序无法直接访问内核数据结构，必须通过系统调用接口。
:::

::: tip 重难点解析
**页表（Page Table）结构 — 虚拟地址到物理地址的翻译机制**

虚拟内存的核心是页表——一个由硬件 MMU（Memory Management Unit）用来将虚拟地址翻译为物理地址的数据结构。

**x86-64 的四级页表（4-Level Paging）**：

在 64 位 Linux 上，使用 48 位虚拟地址（高 16 位必须是全 0 或全 1，用于 canonical address 检查），页大小为 4KB（默认）。48 位地址被分为 5 个字段：

```
| [63:48] 未使用 | [47:39] PGD | [38:30] PUD | [29:21] PMD | [20:12] PTE | [11:0] Offset |
    16 bits        9 bits        9 bits        9 bits        9 bits       12 bits
```

翻译过程（由 MMU 硬件自动完成，每次内存访问都会执行）：

1. **PGD (Page Global Directory)**：从 CR3 寄存器获取 PGD 基地址，用虚拟地址 [47:39] 位作为索引查找
2. **PUD (Page Upper Directory)**：从 PGD 表项中获取 PUD 基地址，用 [38:30] 位索引
3. **PMD (Page Middle Directory)**：从 PUD 表项中获取 PMD 基地址，用 [29:21] 位索引
4. **PTE (Page Table Entry)**：从 PMD 表项中获取 PTE 基地址，用 [20:12] 位索引
5. 最终的 PTE 包含物理页帧号（PFN），加上 Offset [11:0] 形成完整物理地址

每一级查询都是一次内存访问，所以四级页表在最坏情况下需要 4 次额外内存访问！这就是为什么需要 TLB。

**大页（Huge Pages）**：

- 2MB 大页：省略了 PTE 级别（三级翻译），Offset 变为 21 位
- 1GB 大页：省略了 PMD 和 PTE 级别（两级翻译），Offset 变为 30 位

大页减少了页表层级和 TLB 占用，适合大数据应用（数据库、JVM 堆等），但可能导致内部碎片（internal fragmentation）。

**页表项（PTE）的组成**：一个 PTE 通常 8 字节（64 位），包含：

- 物理页帧号（PFN）：40-52 位不等
- 标志位：Present (P)、Read/Write (R/W)、User/Supervisor (U/S)、Accessed (A)、Dirty (D)、eXecute Disable (XD) 等
:::

::: tip 重难点解析
**页面错误（Page Fault）的类型与处理流程**

页面错误发生在 MMU 无法完成虚拟地址到物理地址的翻译时。CPU 抛出页错误异常（#PF），内核的页错误处理程序接管。有三种类型：

**1. 次要页错误（Minor Page Fault）**：
页面已在物理内存中，但当前进程的页表未映射。例如：

- 进程首次访问一个 mmap 的区域，但页面尚未真正分配（demand paging）
- 多进程共享同一物理页（如共享库 libc），但新进程的页表尚未建立映射

处理：内核仅需更新页表项，设置 PTE 指向已存在的物理页。开销极小（微秒级）。

**2. 主要页错误（Major Page Fault）**：
页面不在物理内存中，必须从磁盘读取。例如：

- 文件映射的页面尚未从磁盘加载
- 页面被换出（swapped out）到交换分区

处理：内核从磁盘读取页面到物理内存 → 更新页表。开销极大（毫秒级，涉及磁盘 I/O）。

**3. 无效页错误（Invalid Page Fault → SIGSEGV）**：
进程访问了非法地址。例如：

- 解引用 NULL 指针
- 访问未映射的地址
- 写入只读页面（写时复制保护）

处理：内核向进程发送 `SIGSEGV` 信号。默认行为是终止进程并生成 core dump。

**Demand Paging（按需调页）**：现代 OS 采用惰性加载策略——`mmap` 分配虚拟地址空间时并不立即分配物理内存，而是在首次访问时触发 minor page fault 才分配。这极大节省了物理内存，因为程序通常只使用其虚拟地址空间的一小部分。
:::

::: tip 重难点解析
**`mmap` 的内部机制 — MAP_PRIVATE vs MAP_SHARED**

`mmap` 不仅用于映射文件，还是 `malloc` 的底层实现（对于大块分配，glibc 使用 `mmap` 而非 `sbrk`）。

**MAP_PRIVATE（写时复制，Copy-on-Write）**：
对映射区域的写入不会修改底层文件或与其他进程共享。内核使用 CoW 技术：

1. 初始映射时，所有进程共享相同的物理页，页表项标记为只读
2. 某进程写入时 → 触发页错误
3. 内核创建该页的私有副本，更新该进程的页表指向新副本
4. 其他进程继续共享原始页面

```c
// MAP_PRIVATE 示例：修改不影响文件和其他进程
void *addr = mmap(NULL, 4096, PROT_READ | PROT_WRITE,
                  MAP_PRIVATE, fd, 0);
strcpy(addr, "modified");  // 只会修改私有副本，不写回文件
```

**MAP_SHARED**：
所有映射同一区域的进程共享同一物理页面，修改对所有人可见并最终写回文件。这是实现共享内存（shared memory）IPC 的基础。

```c
// MAP_SHARED 示例：父子进程通信
void *addr = mmap(NULL, 4096, PROT_READ | PROT_WRITE,
                  MAP_SHARED | MAP_ANONYMOUS, -1, 0);
if (fork() == 0) {
    strcpy(addr, "hello from child");  // 子进程写入
} else {
    waitpid(-1, NULL, 0);
    printf("%s\n", (char *)addr);  // 父进程读取到子进程写入的内容
}
```

注意 `MAP_ANONYMOUS` + `MAP_SHARED` 的组合：不与文件关联，但父子进程共享同一物理页。这是最轻量的进程间通信方式之一——不需要管道、不需要信号，直接共享内存。
:::

# 上下文切换

* 为了让所有进程看起来像是同时运行的，内存管理单元（Memory Management Unit，MMU）和进程调度器（process scheduler）协同工作，给每个进程分配一系列短时间片，让其访问物理 RAM，并将进程暂停以支持其他进程访问。
* 当一个进程被挂起以支持另一个进程时，内核必须将其寄存器值（general-purpose registers、程序计数器（program counter）、栈指针（stack pointer）等）保存到 RAM 中以便后续检索。
* 类似地，当进程重新活跃时，内核必须将之前保存的值重新加载到所有寄存器中，以便进程可以从上次暂停的地方恢复执行，就像什么也没发生一样。
* 这个进程的挂起和恢复称为上下文切换（context switch）。

::: tip 重难点解析
**上下文切换的开销**：上下文切换不是免费的。每次切换都需要：(1) 保存当前进程的寄存器状态到内存；(2) 刷新 TLB（部分架构）和 CPU 缓存；(3) 加载下一个进程的寄存器状态。这些操作消耗 CPU 周期而不做任何"有用工作"。因此，频繁的上下文切换会显著降低系统吞吐量。这也是为什么在设计并发程序时要避免不必要地创建过多进程/线程——过多的并发实体导致过多的上下文切换，反而降低整体性能。在 CS111 中你会学习到"调度开销"的量化分析。

**上下文切换的触发条件**：

- 时间片耗尽（定时器中断）
- 进程执行 I/O 操作并阻塞
- 进程主动让出 CPU（`sched_yield()` 或 `sleep()`）
- 更高优先级的进程变为就绪状态（抢占式调度）
:::

::: tip 重难点解析
**TLB（Translation Lookaside Buffer）— 地址翻译的高速缓存**

页表存储在内存中，每次内存访问都可能需要 4 次页表查询（对于四级页表）。如果没有缓存，这意味着每次内存访问都要额外做 4 次内存访问——性能灾难。

**TLB 是什么**：

TLB 是 CPU 内部的一个小型、高速的硬件缓存，缓存最近使用的虚拟地址到物理地址的翻译结果。它本质上是一个**全相联（fully-associative）或组相联（set-associative）缓存**，通常只有几十到几百个条目。TLB 命中（hit）时，翻译仅需 1 个时钟周期。TLB 未命中（miss）时，MMU 需要遍历页表（称为 page walk），可能花费数十到数百个周期。

**TLB 与上下文切换**：

每个进程有自己独立的虚拟地址空间 → 不同的页表 → TLB 中的条目在切换后全部失效。传统做法是**刷新 TLB**（TLB flush）：将所有 TLB 条目标记为无效。下一个进程开始时面临"冷 TLB"——大量 TLB miss，严重影响性能，这就是"上下文切换的间接成本"的主要来源。

**PCID（Process-Context Identifier）优化**：

现代 x86-64 支持 PCID（在 Linux 中使用 PTI/KAISER 补丁后广泛启用）。TLB 的每个条目带有 PCID 标签，标识属于哪个进程。上下文切换时**不需要**刷新 TLB——只需切换到新的 PCID。新进程执行时会逐渐淘汰旧 TLB 条目。

```bash
# 查看 CPU 是否支持 PCID
grep pcid /proc/cpuinfo
```

**ASID（Address Space Identifier）**：ARM 架构的等价概念，称为 ASID。

**TLB 覆盖范围（TLB Reach）**：
假设 64 个 TLB 条目，每页 4KB → TLB 覆盖范围 = 64 x 4KB = 256KB。如果程序的工作集超过 256KB，就会频繁 TLB miss。使用 2MB 大页：64 x 2MB = 128MB 覆盖范围，大幅减少 TLB miss。这就是为什么数据库和 JVM 等内存密集型应用推荐使用大页。
:::

::: tip 重难点解析
**上下文切换的真实成本 — 直接成本 vs 间接成本**

上下文切换的开销不能只看保存/恢复寄存器的几个指令。真正的成本分为两类：

**直接成本（内核执行时间）**：

- 保存/恢复寄存器（通用寄存器、PC、SP、浮点寄存器等）：~几十 CPU 周期
- 切换页表（写 CR3 寄存器）：~几十 CPU 周期
- TLB 刷新（如无 PCID）：~几百到几千 CPU 周期（TTL miss 代价）
- 内核调度代码执行（选下一个进程）：~几百 CPU 周期

典型总直接成本：**几千 CPU 周期**（微秒级）

**间接成本（缓存污染，Cache Pollution）**：
这是上下文切换隐藏的、更大的成本。进程 A 运行时在 CPU 的 L1/L2/L3 缓存中积累了大量热数据。切换到进程 B 后：

- 进程 B 的数据不在缓存中 → 海量 cache miss → 从内存/L3 加载 → 大量额外延迟
- 进程 A 的热数据被逐出缓存 → 当进程 A 再次切换回来时，也需要重新加载

间接成本可达数万到数十万 CPU 周期，是直接成本的 10-100 倍。

**为什么线程比进程轻量？**

|                   | 进程切换                         | 同进程线程切换               |
| ----------------- | -------------------------------- | ---------------------------- |
| 保存/恢复寄存器   | ✓ 需要                          | ✓ 需要                      |
| 切换页表 (写 CR3) | ✓ 必须                          | ✗ 不需要（共享 mm_struct）  |
| TLB 刷新          | ✓ 部分需要                      | ✗ 不需要                    |
| 缓存影响          | 大（不同地址空间，缓存完全替换） | 小（共享地址空间，缓存复用） |

这就是 `pthread_create` 比 `fork` + `execvp` 快得多的根本原因——线程创建不需要复制页表、不需要分配新的 mm_struct、不需要加载新的程序映像。
:::

# 调度

* 调度器负责监督所有进程并决定下一个应该运行的进程。
* 正如任何好的调度器一样，它会通过选择当前活跃但当前未被阻塞（即等待用户输入、等待管道中数据到达、响应信号等）的进程来最大化吞吐量。
* 它通过维护一系列就绪队列（run queues）或运行中队列来实现这一点，进程在被选中执行之前都在这些队列中。
* Linux 调度器实现了完全公平调度器（Completely Fair Scheduler，CFS）算法，使用红黑树（red-black tree）来存储进程，键值为虚拟运行时间（vruntime）。这种红黑树结构使得选择 vruntime 最小的进程和线性组织进程都非常快速。

::: tip 重难点解析
**CFS 与红黑树**：Linux 的 CFS 调度器不使用传统的"时间片"概念，而是追踪每个进程的 vruntime（虚拟运行时间）——进程实际获得的 CPU 时间除以进程的权重（优先级）。CFS 总是选择 vruntime 最小的进程运行（即"获得 CPU 时间最少"的进程），并通过红黑树高效维护这个排序。红黑树是一种自平衡二叉搜索树，插入、删除和查找操作均为 O(log n)，非常适合调度器频繁更新的场景。

**CS111 衔接**：在 CS111 中，你将从更理论的角度学习调度算法：先来先服务（FCFS）、最短作业优先（SJF）、优先级调度、多级队列等。CS110 侧重于让你理解"调度器存在的意义"——即在多进程环境中如何公平且高效地分配 CPU 资源。
:::

# 进程的生命周期

* 进程是处于五个状态之一的程序的一个实例。

  * **就绪（Ready）**：进程已经准备好运行，只是需要 CPU。
    它已经加载了需要加载的所有东西，但只是被暂停了。
    从就绪状态，它可以通过被调度器选中变为**运行中**。
  * **运行中（Running）**：进程正在执行指令。从运行中状态，
    它可以通过等待某个资源变为空闲而变为**阻塞**。
    它可以通过被中断而变为就绪。如果它决定退出，
    它可以变为**终止**。
  * **阻塞（Blocked）**：进程正在等待某个资源，如果它获取到
    该资源，就重新变为**就绪**状态。
    如果它收到 SIGKILL 信号，也可以变为**终止**状态。
  * **终止（Terminated）**：进程已完成。一旦被终止，进程就不会再复活。
  * **挂起（Suspended）**：进程在内存上不会活动。它进入此状态
    以释放内存资源。进程通过其自身或另一个进程从**阻塞**或**就绪**状态进入挂起状态。
    它重新被激活时（通过其自身或通过管理进程），返回到**阻塞**或**就绪**状态（取决于它
    来自哪个状态）。

  ```

                          <--------- 挂起 (Suspended) --------->
                          |                                    ^
                          v                                    |
    新建 (New) --> 就绪 (Ready) <===> 运行中 (Running) --> 终止 (Terminated)
                      ^                  ^    |
                      |                  |    |
                      |    阻塞 (Blocked) |    |
                      +--------->---------+----+
                                        |
                                        v
                                    挂起 (Suspended)
  ```

::: tip 重难点解析
**五状态进程模型**：这是操作系统中最基础的概念模型之一。

1. **新建 → 就绪**：进程被创建后，OS 为其分配 PCB（进程控制块）和必要的资源，然后将其放入就绪队列。
2. **就绪 ↔ 运行中**：调度器选择就绪进程运行（dispatch），或运行中的进程时间片耗尽被抢占（preempt）。
3. **运行中 → 阻塞**：进程发起 I/O 操作或等待某个事件（如 `waitpid`），主动放弃 CPU。
4. **阻塞 → 就绪**：等待的事件完成（I/O 完成、信号到达），进程重新变为可运行状态。
5. **运行中 → 终止**：进程正常退出（`exit`）或被信号杀死（`SIGKILL`）。
6. **挂起状态**：当系统内存不足时，OS 可能将某些进程换出（swap out）到磁盘，这就是"挂起"状态。这是中等调度（medium-term scheduling）的一部分。

在 CS111 中，你还会学习到"七状态模型"——将挂起状态细分为"挂起就绪"和"挂起阻塞"两种子状态。
:::

# printf 能工作吗？

* 那些熟悉并发编程的人会认为：在多线程的世界中，`printf` 可能不是线程安全的（大多数函数不是），因为在没有额外同步的情况下，
  多个线程的并发 `printf` 调用可能会导致控制台出现字符
  交织的混乱输出。
  * 但我们可以——在信号处理器内使用 `printf`——事实上，
    它在信号处理器上下文中是线程安全的吗？
  * 也许不是。确实不是。实际上，绝对不是，但幸运的是，有一组有限的函数，
    被标准 POSIX 指定为*异步信号安全（async-signal-safe）*的，
    这意味着它们可以从信号处理器中被安全地调用。
  * `printf` 不在该列表中，但 `write` 在。
    `printf` 不能安全使用的主要原因是它对输出流进行缓冲

    （回想一下，所有 C 库函数都假定单线程环境，并使用具有持久生命周期的全局变量数组）。
  * 感兴趣的同学可以参考教材的 8.5.5 节以获取异步信号安全
    函数的详尽列表。

::: tip 重难点解析
**异步信号安全（Async-Signal-Safety）**：这是一个容易被忽视但极其重要的概念。信号处理器可能在任意时刻"中断"主程序的执行——包括主程序正在执行 `printf` 的时刻。如果信号处理器中也调用 `printf`，两者会同时操作 `stdout` 的内部缓冲区，导致数据竞争（data race）和缓冲区损坏。

POSIX 标准定义了一组"异步信号安全"函数，保证在信号处理器中调用是安全的。这些函数通常是不使用全局状态的简单系统调用包装，如 `write`、`read`、`_exit`、`waitpid` 等。一个实用的经验法则：**在信号处理器中只使用 `write` 进行输出，绝对不要使用 `printf`、`malloc` 或任何可能持有锁的函数**。在 CS111 中，你会学习到线程安全（thread-safety）的更全面讨论，包括可重入函数（reentrant function）和互斥锁（mutex）的使用。
:::
